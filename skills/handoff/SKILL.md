# 生成交接

## 目标

把当前对话收敛成下一会话可直接接手的工程交接，并写入 `.codex/handoff.md`。handoff 只记录继续推进必需的事实，不是主线程绕过收尾、补证或委派要求的许可；`无需委派` 只表示写交接前无需额外补查，不恢复主线程自执行权限。

## 停止门槛：先停下并委派

- 在打开仓库文件、运行 `rg`/`git`/测试命令、核对 diff、补查入口命令、拆分任务或整理代码事实之前，先判断当前对话是否已经足够生成 handoff。
- 若当前仍有可委派的补查、收尾、验证、任务拆分或执行前准备工作，且当前环境允许委派，必须先委派，再继续生成 handoff；即使当前对话与现有证据已经足够写出 handoff，也不得跳过这一步。主线程只负责派发、审核和收敛，不得先亲自补查仓库再决定是否委派。
- `没有看到 subagent tool`、`这里只有并行命令`、`我只想快速看一眼仓库`、`上下文快满所以我先自己查` 都不是例外理由。
- 只有在明确不可委派时，主线程才可做最小范围补查；并且必须同时记录：
  1. 原因：具体是哪条更高优先级限制、平台能力缺失或会话约束阻止委派。
  2. 证据：已核实哪些工具或约束，因此确认无法委派。
  3. 最小范围：主线程只补查哪些事实，为什么这是生成 handoff 的最低必要集合。
  4. 影响与回退点：因此带来的风险，以及何时恢复委派。
- 缺少任一项例外证明时，立即停止，不得继续主线程补查仓库。

## 核心规则

- 当前对话是第一信息源；只有在继续推进所必需时才补查仓库。
- handoff 可以记录未完成工作，但不能掩盖停止点。若用户要求现在停止，或当前轮确实无法继续推进，可以生成 unfinished handoff；但必须明确写出停止点、未完成项、阻塞和下一步，不能假装本轮已收尾。
- 主线程只保留协调、审核和收敛所需的最小上下文；历史细节、大段日志、diff 和实现细节默认裁剪，或交给子代理压缩后再接收摘要。
- 缺失信息只做最小必要推断，并标注 `（推断）`；不得虚构改动、命令、测试或结果。
- 若仓库 `AGENTS.md` 会影响恢复后的执行方式、角色分工或风险边界，handoff 必须提炼 1-3 条最关键、可执行的约束。
- `下一步任务` 必须写成可执行任务。若当前仓库或当前对话要求主线程协调、子代理执行，或对子代理模型/推理强度有明确约束，handoff 必须原样继承这些约束，不得自行扩写、改写或省略。
- handoff 负责持久化停止点、约束与委派状态，不负责定义恢复后的运行算法；但若恢复后仍适用主线程协调、子代理执行的规则，必须把 `恢复后首动作` 写成可执行事实，例如 `派发子代理`、`续派`、`等待已有子代理`、`收敛回包`，即使 handoff 时没有活跃委派也一样。
- `恢复后首动作` 用于持久化恢复协议，不得写成“继续推进”“继续处理”这类模糊表述；只有在当前仓库与当前对话都没有协作分工约束时，才可写 `无明确协作动作（推断）`。
- 若当前轮已有活跃委派，handoff 必须把委派状态写成可执行事实：谁在执行、执行域、主线程等待期允许动作、解除等待或接管的条件。不得只写“继续推进”“继续修改文档”这类模糊描述。
- 后续用户确认、批准或催继续，只批准范围、顺序或风险处理，不改变 handoff 已记录的执行者分配；除非出现新的更高优先级指令明确改写协作关系。
- 默认紧凑模式：少于 10 KB，建议 2-4 KB。`计划` 保持 2-4 项；`关键改动文件`、`已验证`、`未验证`、`风险`、`下一步任务` 默认各不超过 3 项。

## 流程

1. 判断当前轮是否已到可说明的停止点；若仍需继续推进，先决定是继续当前轮，还是明确停止点、未完成项和阻塞后再 handoff。若还需要补证或执行性工作，必要时先委派。
2. 判断当前对话是否已足够生成 handoff，并单独判断停止前是否仍有可委派的补查、收尾、验证、任务拆分或执行前准备工作；只要这类工作仍存在，就先委派，不能因为“已经足够写 handoff”而跳过。只有满足“不可委派例外证明”时，主线程才可做最小范围补查。
3. 从当前对话和必要证据提取目标、阶段、计划、阻塞点、关键改动、验证状态、风险、下一步，以及会影响恢复执行的最小协作约束与 `恢复后首动作`。
4. 用固定结构生成交接；未知信息直接写 `未知` 或 `暂无明确约束（推断）`。
5. 写入 `<git-root>/.codex/handoff.md`；如果当前目录不在 Git 仓库内，则写入 `<cwd>/.codex/handoff.md`。若 `.codex/` 不存在，先创建。

## 输出结构

【目标】
- 功能
- 当前阶段
- 计划

【关键约束】

【委派状态】
- 无需委派：当前对话与现有证据已足够写出 handoff，无需额外补查；不表示主线程可恢复自执行
- 恢复后首动作：派发子代理 / 续派 / 等待已有子代理 / 收敛回包 / 无明确协作动作（推断）
- 或已委派：任务、执行域、执行者、主线程等待期允许动作、解除等待/接管条件
- 或例外：原因、证据、最小范围、影响、回退点

【当前状态】
- 已完成
- 未完成
- 当前阻塞点
- 关键改动文件
- 已验证
- 未验证
- 风险

【核心实现上下文】（按需；仅保留 1-3 条恢复执行必需事实）

【关键代码】（默认省略）

【运行与验证】
- 入口命令
- 构建/部署方式
- 验证方法

【下一步任务】

【执行要求】
- 禁止整体重写系统
- 必须基于现有代码推进
- 修改必须说明影响
- 优先解决阻塞问题
- 所有结果必须可验证
- 主线程恢复后继续保持最小上下文：默认只依赖 `.codex/handoff.md` 与必要证据
- 若下一步需要显式角色分工，写清执行者与审核者
- 若存在协作分工或对子代理模型/推理强度的明确约束，按当前仓库或当前对话原样写入 handoff
- 用户后续确认只改变范围、顺序或风险处理，不单独改变执行者分配；除非新的更高优先级指令明确覆盖

## 裁剪顺序

1. 删除 `关键代码`。
2. 将 `核心实现上下文` 压到 1-3 条恢复必需事实。
3. 删除重复说明与低价值背景。
4. 将文件列表缩到最关键 1-3 项。
5. 压缩历史性交接，只保留当前推进必需内容。

不得删除 `委派状态`、`当前阻塞点`、`运行与验证`、`下一步任务`、`执行要求`。

## 自检

- 生成 handoff 前，我已经明确当前是“已收尾”还是“停在可说明的停止点”，没有用 handoff 掩盖本轮结论。
- 若仍需补证、验证、任务拆分或执行性 Git 工作，我先委派；若没有委派，已写出完整例外证明。
- 我没有把 `无需委派` 写成主线程恢复自执行的许可；若恢复后仍需遵守委派优先规则，已明确写出 `恢复后首动作`。
- 我已把恢复执行必需的协作分工、执行域、等待条件、首动作与模型约束原样写入 handoff。
