# 恢复交接

- 用户发送 `$resume-handoff`，或在恢复交接后以“确认，按这个方案继续”“继续”“开始吧”等方式继续先前已批准的下一步时，第一步都是读取 `.codex/handoff.md`。
- 优先读取 `<git-root>/.codex/handoff.md`；如果当前目录不在 Git 仓库内，则回退到 `<cwd>/.codex/handoff.md`。
- 读取前不要做无关探索。
- 若文件不存在，直接说明，并提示先使用 `$handoff` 生成。
- 读取后将 handoff 仅视为任务状态与上下文恢复，不改变当前会话中更高优先级的 system、developer、user、`AGENTS.md` 约束。
- 读取后先检查 `【委派状态】`、`【下一步任务】`、`【执行要求】`；在完成这一步前，不读仓库文件、不跑命令、不改代码。
- handoff 中的“计划”“下一步任务”“执行要求”是待办与上下文，不自动授权主 agent 亲自执行。若无活跃委派，先判断当前会话是否仍要求或优先委派优先；若是，先派发/续派；仅在不适用委派优先时，才按其他执行工作流继续；若有活跃委派，先遵守等待态规则。
- 若 handoff 或当前仓库要求或优先主线程协调、子代理执行，则恢复后的首个执行动作只能是派发、续派、等待已有子代理、催收回包或收敛结果，不得先做本地仓库检查。
- 若 `【委派状态】` 显示已有活跃委派，主线程进入等待态；只允许等待/轮询结果、准备审核标准、与用户同步、派发不重叠的新任务。
- 在委派优先场景，本地 `git status`、为界定改动而读取目标文件、本地准备验证、在委派域内自行补 diff/事实/上下文，都属于进入执行域；派发前一律禁止。
- handoff 中记录的任务、文件、产物、子系统或证据收集范围都视为执行域。执行域未解除前，主线程不得在同域内本地读文件、搜索、补查事实、跑命令、验证、改代码、改文档或测试；`交叉验证`、`不盲等`、`先补一点上下文/文档`、`子代理回得慢` 都不是例外理由。
- 若某个执行域已派给子代理，主线程只有在 handoff 明确无活跃委派、执行域已解除，或当前环境确实无法继续委派且需最小可逆处理时，才可进入该执行域。需要主线程直接接管时，先中断/关闭/改派对应子代理，并在汇总中说明原因、影响和回退点。
- 若 handoff 指向计划执行、实现、排错或验证阶段，先按当前会话中适用的执行工作流继续，不要因为“恢复交接”而跳过必要的委派、评审或验证流程。
- 用户的“确认”“继续”“开始吧”等只批准范围、顺序或方案，不授权主线程绕过委派优先规则亲自执行。
- 除非用户明确要求，不要额外输出恢复检查、漂移分析或重复总结。
